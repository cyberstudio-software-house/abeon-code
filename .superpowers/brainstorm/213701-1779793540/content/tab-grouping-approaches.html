Tab Grouping — Implementation Approaches

All approaches use collapsible group headers, scroll arrows on overflow, hidden header for single project, and "ProjectName (N)" for collapsed groups. The difference is in how group state and rendering are structured.

A

Pure Derived State (Recommended)

How: Group headers and ordering are computed on render via a useMemo that groups tabs by projectId. Collapsed state lives in a local useState<Set<number>> inside TabBar.

Pros

  • Zero store changes — tabs array stays flat
  • Tab persistence unchanged
  • Collapsed state resets on app restart (clean slate)
  • Simplest to implement and review

Cons

  • Collapsed state lost on component unmount (negligible — TabBar is always mounted)
  • If we later want persisted collapse state, needs migration to store
B

Store-Level Grouping

How: Add collapsedProjectIds: Set<number> to tabsSlice with a toggleGroupCollapsed(projectId) action. Grouping logic still computed on render, but collapse state persists across the session via store.

Pros

  • Collapsed state survives any re-renders
  • Easy to persist to localStorage later if needed
  • Other components could react to collapse state

Cons

  • Adds state to the global store for a UI-only concern
  • Slightly more code in the slice
C

Virtual Scroll with Groups

How: Flatten groups + tabs into a virtual list. Each item is either a group header or a tab. Use a virtualized horizontal list (e.g., react-window) so only visible items render.

Pros

  • Handles 100+ tabs with zero perf penalty
  • Most scalable solution

Cons

  • Adds a dependency (react-window or similar)
  • Horizontal virtualization is tricky with variable-width tabs
  • Overkill — AbeonCode is unlikely to have 100+ tabs
  • Most complex implementation by far
