# ClickUp Integration (Design)

**Date:** 2026-06-30
**Status:** Approved (design) — pending spec review before planning
**Scope of this document:** A new ClickUp integration in the AbeonCode DesktopApp:
connection config, a per-project "Zadania ClickUp" right-panel section, task
linking + detail view, a file-based task "handle" the agent can read, a
model-generated work summary posted as a ClickUp comment, and (final phase) a
work-time estimate written to the task.

## Goal

Let a developer working in AbeonCode tie a project to its ClickUp tasks and pull
those tasks into the AI-CLI workflow:

- Configure a ClickUp connection once in settings.
- See a "Zadania ClickUp" section in the right panel (between Actions and Git).
- Link tasks to the current project by searching by ID/URL or by name.
- Open a linked task to read its name, description, attachments, and comments.
- Get a per-task **handle** to paste into a session so the agent can read the
  task content.
- Generate a concise summary of completed work from the active session and post
  it as a comment on a chosen linked task.
- (Final phase) Estimate work time for a session and write a proposed value to
  the task.

## Decisions (locked)

1. **Auth:** ClickUp **Personal API Token** (no OAuth). Stored SQLite-side,
   read/written only by Rust, never exposed to the renderer — mirrors the
   `remoteDeviceSecret` precedent. UI sees only a connection status.
2. **API location:** all ClickUp REST calls happen **Rust-side** via a
   `ClickUpClient` built on reqwest 0.12 (already a dependency), cloning the
   shape of `remote/cloud_client.rs::CloudClient`.
3. **Search scope:** **per-project** (workspace → space → optional list).
   Name-search runs within that scope (ClickUp API v2 has no global full-text
   task search); lookup by **ID/URL** works globally via `GET /task/{id}`.
4. **Handle mechanism:** **file in repo + reference.** Task content is written to
   `<project>/.abeon/clickup/<id>.md` (gitignored), regenerated on use/refresh;
   the handle is the relative path (e.g. `@.abeon/clickup/CU-123.md`).
5. **Work summary:** generated by the model from the active session, shown in an
   **editable preview**, then posted as a **comment** on a chosen linked task.
6. **Time estimate (final phase):** active session time (idle gaps trimmed) +
   model-estimated experienced-dev time, blended via an adjustable slider,
   editable, written to the task as a **time-tracking entry**.
7. **Model engine:** summary and time estimate reuse the existing title-gen
   mechanism (`claude -p` / `codex exec --ephemeral`) — no new model integration.
8. **Section placement:** `RightPanel.tsx` renders Actions → Git → Usage; the new
   `ClickUpSection` is inserted **between Actions and Git** →
   Actions → ClickUp → Git → Usage.

The token never crosses the IPC boundary. The renderer triggers actions and
renders typed results; the Rust side holds the secret and talks to ClickUp.

## Data model

New migration `db/migrations/004_clickup.sql` (registered in `db/mod.rs` with
`const MIGRATION_004` + `if v < 4 { ... }`, ending with
`INSERT OR IGNORE INTO schema_version(version) VALUES (4);`).

```sql
CREATE TABLE IF NOT EXISTS clickup_project_config (
  project_id   INTEGER PRIMARY KEY REFERENCES projects(id) ON DELETE CASCADE,
  workspace_id TEXT NOT NULL,
  space_id     TEXT,            -- scope for name search
  list_id      TEXT             -- optional narrowing to a list
);

CREATE TABLE IF NOT EXISTS clickup_links (
  project_id INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  task_id    TEXT NOT NULL,
  custom_id  TEXT,              -- human id (e.g. CU-123) when present
  name       TEXT NOT NULL,     -- cached name for the list row
  status     TEXT,              -- cached status chip
  url        TEXT NOT NULL,
  linked_at  INTEGER NOT NULL,
  PRIMARY KEY (project_id, task_id)
);
CREATE INDEX IF NOT EXISTS idx_clickup_links_project ON clickup_links(project_id);
```

Repos in `db/`: `clickup_config_repo.rs` (UPSERT by `project_id`, like
`session_titles_repo`) and `clickup_links_repo.rs` (list/get/insert/delete by
`project_id`, like `actions_repo`).

The token lives in the existing `settings` key/value table under key
`clickupApiToken`, written by a dedicated command, **not** added to
`PERSISTED_KEYS` and **not** present in `settingsSlice` — so it never reaches the
Zustand store or localStorage.

## Backend: ClickUpClient + commands

`clickup/` module — `ClickUpClient { http: reqwest::Client, token: String }`,
base `https://api.clickup.com/api/v2`, `bearer_auth(token)` on every call.
Typed responses (ts-rs domain types). Error enum (thiserror) maps:
`401 → InvalidToken`, `429 → RateLimited`, transport error → `Offline`,
other non-2xx → `Api { status, message }`. Pattern cloned from `CloudClient`.

ClickUp REST endpoints used:
- `GET /user` — connection test.
- `GET /team` — workspaces; `GET /team/{id}/space` — spaces;
  `GET /space/{id}/list` + `GET /folder/{id}/list` — lists (for the scope picker).
- `GET /task/{id}` (with `custom_task_ids=true&team_id=<workspace>` when resolving
  a custom id like `CU-123`) — task by ID/URL and full detail.
- Name search (results filtered locally by name, case-insensitive):
  - if `list_id` is configured → `GET /list/{list_id}/task` (paginated);
  - else if only `space_id` → `GET /team/{workspace_id}/task?space_ids[]=<space_id>`
    (filtered team tasks, paginated).
- `GET /task/{id}/comment` — comments. Attachments come back on the task detail.
- `POST /task/{id}/comment` — post work summary.
- `POST /task/{id}/time_entries` — write the time estimate (final phase).

`commands/clickup.rs` (registered in `lib.rs` `generate_handler!`, wrappers in
`src/lib/tauri.ts`, args camelCase):
- `clickup_set_token(token)`, `clickup_clear_token()`,
  `clickup_connection_status() -> ClickUpConnectionStatus` (configured/invalid/absent).
- `clickup_list_workspaces() / clickup_list_spaces(workspaceId) / clickup_list_lists(spaceId)`.
- `clickup_get_config(projectId) / clickup_set_config(projectId, workspaceId, spaceId?, listId?)`.
- `clickup_search_tasks(projectId, query) -> Vec<ClickUpTaskRef>` (ID/URL → direct
  GET; otherwise scoped name search).
- `clickup_link_task(projectId, taskId) / clickup_unlink_task(projectId, taskId)`,
  `clickup_list_links(projectId) -> Vec<ClickUpLink>`.
- `clickup_get_task(taskId) -> ClickUpTaskDetail` (name, description, status,
  attachments, comments).
- `clickup_write_task_file(projectId, taskId) -> String` (returns relative path).
- `clickup_generate_summary(projectId, sessionId, provider) -> String`.
- `clickup_post_comment(taskId, text)`.
- `clickup_estimate_time(projectId, sessionId, provider) -> TimeEstimate` (final phase).
- `clickup_log_time(taskId, durationMs, description?)` (final phase).

All network commands are `pub async fn (state: State<'_, AppState>, ...)`.

New ts-rs domain types (`domain/clickup.rs`, exported to `src/types/`):
`ClickUpConnectionStatus`, `ClickUpWorkspace`, `ClickUpSpace`, `ClickUpList`,
`ClickUpTaskRef`, `ClickUpTaskDetail`, `ClickUpComment`, `ClickUpAttachment`,
`ClickUpLink`, `ClickUpProjectConfig`, `TimeEstimate`. Run `cargo test` once to
materialize the `.ts` files.

## Frontend

`store/clickupSlice.ts` — per-project state: `linksByProject`,
`configByProject`, `connectionStatus`, loading/error flags, and loaders that call
the `tauri.clickup*` wrappers (consistent with `gitSlice`/`actionsSlice`).

### Settings — "ClickUp" tab (`SettingsDialog.tsx`)

New `SettingsTab` value `'clickup'` + `<TabButton>` + body switch + a
`ClickUpTab()` component co-located in the file. Contents: masked token input
(write via `clickup_set_token`, never read back), **"Testuj połączenie"** button
(`GET /user`), status line, "Usuń token". No token value is ever held in the
store.

### Right panel — `right/ClickUpSection.tsx`

Inserted in `RightPanel.tsx` between `<ActionsSection/>` and `<GitSection/>`
(plus a divider). Standard section header (`text-[10px] uppercase tracking-wider`
"ZADANIA CLICKUP") with a small settings affordance (scope: workspace/space/list
pickers) and a link count. Body: list of linked tasks from `clickup_links`
(row = `custom_id`/short id chip + name + status chip), row click → detail popup,
hover → "Odepnij". A `+ powiąż zadania` text button (like `ActionsSection`'s
`+ dodaj akcję`) opens the link dialog via local `useState`. Empty states for
missing token / missing scope.

### Link dialog — `dialogs/LinkClickUpTaskDialog.tsx`

Standard `fixed inset-0` overlay; Sidebar-style search input. If the input looks
like an ID/URL → direct `GET /task/{id}`; otherwise debounced scoped name search
(`useEffect` on `query`). Results = clickable rows; click → `clickup_link_task`,
already-linked rows marked. Closes on overlay click / Escape.

### Task detail popup — `dialogs/ClickUpTaskDialog.tsx`

Cloned from `DiffDialog`'s shell (bordered header + close). Header: custom_id +
name + "otwórz w ClickUp" link. Body: description (markdown render), status /
assignee, attachments (clickable links opened in the default browser), comments
(read-only list). Fetch via `clickup_get_task` in a `useEffect` keyed on
`taskId` with `loading/error`/cancel-flag. Action bar: **Kopiuj uchwyt**,
**Wstaw do aktywnej sesji**, **Odśwież**, **Generuj podsumowanie**, **Odepnij**.

### Handle / file mechanism

`clickup_write_task_file` fetches fresh detail, formats markdown (name,
description, comments, attachment links), writes
`<project.path>/.abeon/clickup/<id>.md`, ensures `.abeon/` is in the project's
`.gitignore` (appends if missing), and returns the relative path. The handle is
that path with an `@` prefix (Claude-friendly), e.g. `@.abeon/clickup/CU-123.md`.
"Kopiuj uchwyt" copies it to the clipboard; "Wstaw do aktywnej sesji" writes it
into the active session's terminal from the frontend (the `TerminalView`
image-paste path that already injects a file path into the PTY — provider-
agnostic, works for Claude and Codex), without a trailing `\r` so the user can
keep typing. The file is (re)generated on "Wstaw" and "Odśwież" — always a fresh
snapshot.

> Frontend injection requires plumbing the active session's pty id out of
> `TerminalView` (today it is local to that component; only action PTYs are in
> the store). The backend `session_pty` registry is Claude-only, so it is not
> used for this. Copy-to-clipboard is the provider-agnostic fallback that always
> works.

### Work summary — `dialogs/ClickUpSummaryDialog.tsx`

Triggered for the **active session** (active tab → `sessionId` + `provider`).
`clickup_generate_summary` reuses `history_blocks_for_session` (cap 500),
flattens like `render_markdown`, builds a "concise summary of completed work for
ClickUp" prompt, and calls the model via the title-gen path (timeout +
`kill_on_drop`). UI: editable preview textarea + target-task dropdown (from
`clickup_links`) → "Wyślij jako komentarz" → `clickup_post_comment`.

### Time estimate (final phase) — `dialogs/ClickUpTimeDialog.tsx`

`clickup_estimate_time` returns: **active session time** = sum of inter-block
`timestamp` deltas with idle gaps over a threshold (default 5 min, configurable)
trimmed; **dev estimate** = model estimate of experienced-dev time for the
delivered work. UI shows both + a blend slider →
`proposed = session + k·(dev − session)`, editable. "Zapisz czas" →
`clickup_log_time` posts a `POST /task/{id}/time_entries` entry on the task.

## Error handling & edge cases

- **401** invalid/expired token → connection status reflects it; actions surface
  a clear message and point to settings.
- **429** rate-limit (ClickUp ~100 req/min) → user-facing message; light backoff
  on the client.
- **Offline / transport error** → graceful failure in the dialog, no crash.
- **No active session** → summary/time actions disabled with a hint.
- **Task deleted in ClickUp** → refresh/detail shows an error and offers unlink.
- **Project deleted** → `ON DELETE CASCADE` removes config + links.

## Phasing

Each phase becomes its own implementation plan.

1. **Connection foundation** — migration `004`, `clickup_config_repo` +
   `clickup_links_repo`, `ClickUpClient` + wiremock tests, token commands,
   "ClickUp" settings tab (token + connection test). Deliverable: connect & verify.
2. **Tasks in project** — scope config (workspace/space/list pickers),
   `ClickUpSection`, link dialog (ID/URL + name), `clickup_links` persistence,
   detail popup, handle/file + `.gitignore` + copy/inject. Deliverable: link →
   view → paste handle.
3. **Work summary** — generate from active session → editable preview → comment.
4. **Time tracking (final)** — estimate + slider + time entry.

## Testing

- **Rust:** `ClickUpClient` per-endpoint tests with `wiremock` (already a
  dev-dep); command-level tests against the SQLite repos.
- **Frontend:** `vitest` for `clickupSlice` and the dialogs.
- `npm run lint` clean; ts-rs types regenerated via `cargo test`.

## Out of scope (YAGNI)

- OAuth2 flow and multi-account ClickUp connections.
- Global (whole-workspace) full-text task search.
- Inline image preview of attachments (open-in-browser links only).
- Posting ad-hoc comments beyond the generated work summary.
- Two-way sync / editing ClickUp tasks (status changes, assignees) from AbeonCode.
- Committing task content files to the repo (they stay gitignored).
